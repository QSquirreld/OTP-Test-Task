# SQL-2: Последний контакт по кейсу и оплата после него

## Задача
Для каждого кейса, стартовавшего в марте 2026 года, определить последний контакт и проверить, была ли после него положительная оплата в окне 5 дней, а также посчитать сумму таких оплат.

## Что считалось
По каждому `case_id` получены:
- `case_id`
- `account_id`
- `strategy_name`
- `last_contact_dttm`
- `last_channel`
- `last_contact_result`
- `payment_in_5d_flag`
- `payment_amount_5d`

## Подход
1. Отобраны кейсы со `start_date` в диапазоне `2026-03-01` — `2026-03-31`
2. Для каждого `case_id` найден последний контакт через `MAX(event_dttm)`
3. По паре `case_id + last_contact_dttm` восстановлены `last_channel` и `last_contact_result`
4. По `account_id` проверены положительные платежи в окне от даты последнего контакта до `+4 дней` включительно
5. Для каждого кейса рассчитаны:
   - флаг наличия оплаты в окне
   - сумма положительных платежей в этом окне

## Допущение
Окно после последнего контакта реализовано по календарным датам:

`payment_dttm::date BETWEEN last_contact_dttm::date AND last_contact_dttm::date + 4`

То есть учитывается день последнего контакта и ещё 4 следующих календарных дня.

## Что даёт результат
Итоговая таблица позволяет:
- увидеть последний канал коммуникации по кейсу
- посмотреть, приводит ли последний контакт к оплате в коротком горизонте
- сравнивать типы контактов и результаты контакта с точки зрения постконтактной оплаты

## Как интерпретировать результат
- `payment_in_5d_flag = 1` означает, что после последнего контакта по кейсу был хотя бы один положительный платёж в окне 5 дней
- `payment_amount_5d` показывает суммарный объём таких платежей
- если `payment_amount_5d` равно `NULL`, значит положительных платежей в этом окне не было

## Вывод
Этот результат можно использовать как базу для последующего анализа качества коммуникации:
например, сравнить каналы (`call`, `sms`, `email`) и результаты контакта (`RPC`, `Promise`, `No answer`) по их связи с оплатой после последнего касания.
